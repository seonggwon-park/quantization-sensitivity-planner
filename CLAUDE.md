## Research Context and Source-of-Truth Files

For any substantive research, experiment, code, theory, or planning task, inspect the relevant project documents before drawing conclusions.

### Required reading order

1. `docs/experiment_log.md`
   - Always read `Current Working State` and the entries relevant to the assigned task.

2. `docs/formalization_notes.md`
   - Read when the task involves metrics, mathematical definitions, risk, oracle/proxy distinctions, or implementation of those concepts.

3. `docs/COMPRESSION_PLANNING_AGENT_HANDOFF.md`
   - Read the `Agent Start Here` section at minimum.
   - Read additional relevant sections when historical decisions, prior failures, or research claims are involved.

4. Relevant source code, configs, raw result files, and only the prior session reports relevant to the assigned task.

### Authority rules

- Raw result files, configs, and code are the authority for exact numerical values and implementation behavior.
- `docs/formalization_notes.md` is the authority for current mathematical notation and definitions.
- `docs/experiment_log.md` is the authority for integrated experimental status and current active work.
- `docs/COMPRESSION_PLANNING_AGENT_HANDOFF.md` is the authority for historical context and research decision history.
- `docs/session_reports/` contains worker-session evidence and provisional conclusions. It is not canonical until an integrator verifies and incorporates it.

### Session-report rules

For every substantive task, create exactly one new report under:

`docs/session_reports/YYYY-MM-DD_HHMM_<task-type>_<short-slug>.md`

Examples:

- `2026-06-30_2210_experiment_signed-risk-confirmation.md`
- `2026-06-30_2240_theory_margin-audit.md`
- `2026-07-01_1015_code_proxy-fix.md`

Never overwrite another session's report.

Each report must include:

1. Task and hypothesis
2. Files inspected and files changed
3. Exact command, config, seed, checkpoint, split, and output paths when applicable
4. Important metrics only, with baseline comparison
5. Supported conclusion
6. What the result does not establish
7. Adversarial-audit verdict and findings
8. One recommended next action
9. Whether canonical documents need an update

### Parallel-session rules

Worker sessions may create or modify code, configs, result files, and their own session report.

Worker sessions must not directly edit:

- `docs/formalization_notes.md`
- `docs/COMPRESSION_PLANNING_AGENT_HANDOFF.md`
- the integrated sections of `docs/experiment_log.md`

Only an explicitly assigned integrator session may update canonical research documents after reviewing the relevant session reports, raw results, code diffs, and adversarial audits.

## Completion and research-integrity rules

Do not declare a task complete immediately after producing code, an experiment result, a mathematical derivation, or a research conclusion.

Before completion:

1. Run the applicable adversarial audit procedure.
2. Distinguish verified facts, assumptions, estimates, and unresolved issues.
3. For experiments, provide a user-facing result brief in Korean:

   * Question and hypothesis
   * What was changed or tested
   * Exact important metrics and result-file paths
   * Interpretation: what the result supports and what it does not support
   * Main caveats, confounders, or possible failure modes
   * Recommended next action
4. Do not present a result as a general claim unless the audit finds the necessary evidence.
5. If the audit cannot be completed, explicitly state why and mark the task as incomplete.

Use `docs/session_reports/` for the detailed task report. 

## Mandatory close rule
The `/research-close` skill is responsible for creating exactly one new report under `docs/session_reports/`. If the report cannot be created, the task is incomplete.
For every substantive experiment, code change, mathematical derivation, or research-idea decision:

1. Do not declare completion before running `/research-close`.
2. `/research-close` must run the `research-adversary` audit.
3. The final response to the researcher must contain the concise Korean result briefing generated by the close protocol.
4. A task without an audit verdict and a session report is incomplete.
5. Worker sessions must not update canonical documents directly; only an integrator session may update:

   * `docs/experiment_log.md`
   * `docs/formalization_notes.md`
   * `docs/COMPRESSION_PLANNING_AGENT_HANDOFF.md`
